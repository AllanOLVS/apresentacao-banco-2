08 / Auditoria
Política de Auditoria

Rastreamento de acesso

Por que auditar?

Pesquisadores têm acesso a dados sensíveis de pessoas reais. Saber quem acessou e o que visualizou é essencial para investigações forenses e tomadas de decisão.

Auditoria de Acesso
CREATE SERVER AUDIT
Auditoria_Pesquisadores_Gov
TO FILE (FILEPATH='D:\Logs_Acesso\')
WITH (QUEUE_DELAY=1000,
ON_FAILURE=SHUTDOWN);

-- Monitora SELECT em todas as tabelas
ADD (SELECT ON dbo.Boletim_Ocorrencia)
ADD (SELECT ON dbo.Pessoa_Envolvida)
...
Técnicas Utilizadas
🔒

SQL Server Audit — implementação nativa, sem dependências externas.

⛔

ON_FAILURE = SHUTDOWN — se a auditoria falhar, o banco inteiro é bloqueado.

⏱️

QUEUE_DELAY = 1000ms — gravação assíncrona sem impacto no desempenho.

Validação Geográfica

Coordenadas fora do bounding box de São Paulo são capturadas em tabela de auditoria Auditoria_Validacao_Geografica, evitando destinação de recursos públicos para regiões erradas.

lat: -24.00 a -23.35 lng: -46.82 a -46.36
09 / Permissões
Política de Permissão

Menor privilégio & Mascaramento

Princípio do menor privilégio

Pesquisadores possuem acesso apenas de leitura (SELECT). Nenhum INSERT, UPDATE ou DELETE é permitido, preservando a integridade para outros setores públicos.

Implementação passo a passo
-- 1. Login
CREATE LOGIN Pesquisador_Gov
WITH PASSWORD = '***';

-- 2. Usuário no banco
CREATE USER Login_Pesquisador
FOR LOGIN Pesquisador_Gov;

-- 3. Role + Schema
CREATE ROLE Role_Acesso_Leitura;
GRANT SELECT ON SCHEMA::dbo
TO Role_Acesso_Leitura;

-- 4. Atribuir
ALTER ROLE ... ADD MEMBER Pesquisador;
Dynamic Data Masking (DDM)

Pesquisadores estudam padrões geográficos, não precisam ver dados pessoais das vítimas. O DDM embaralha automaticamente campos sensíveis.

ALTER TABLE dbo.Pessoa_Envolvida
ALTER COLUMN nome_pessoa
ADD MASKED WITH
(FUNCTION = 'default()');

-- Leitura: vê tudo mascarado
GRANT UNMASK TO Role_Acesso_Leitura;
Vantagem do Schema

Se uma nova tabela for adicionada, basta vinculá-la ao schema — as permissões são herdadas automaticamente, sem configuração manual.

SchemaRoleDDM
10 / Backup
Política de Backup

Estratégia Full · Diff · Log

💾
Backup Full
Diário · 02:00

Base principal de recuperação. Agendado às 02h — menor movimento operacional. Garante ponto íntegro e confiável dos dados.

BACKUP DATABASE CrimeData_DB
TO DISK = @FileName
WITH COMPRESSION, INIT, STATS=10;
🔄
Backup Diferencial
A cada 3h · início 05:00

Captura apenas alterações desde o último Full. Equilibra atualidade dos dados, tamanho dos arquivos e impacto no servidor.

📋
Backup de Log
A cada 30 min

Registra cada transação (flagrantes, provas, alterações de status). Permite recuperação ponto a ponto — essencial em auditorias judiciais.

BACKUP LOG CrimeData_DB
TO DISK = @FileName
WITH COMPRESSION, INIT;

Problema resolvido

Cargas ETL do Data Warehouse não competem com transações do dia a dia. O primário atende inserções; o secundário atende consultas pesadas em modo Read-Intent.

Configuração técnica
CREATE AVAILABILITY GROUP [AG_CrimeData]
WITH (CLUSTER_TYPE = NONE)
FOR REPLICA ON
N'sql_primario' WITH (
AVAILABILITY_MODE = ASYNCHRONOUS_COMMIT,
FAILOVER_MODE = MANUAL,
SEEDING_MODE = AUTOMATIC
),
N'sql_secundario' WITH (
SECONDARY_ROLE(ALLOW_CONNECTIONS = ALL)
);
Validação em tempo real

90% da base histórica carregada inicialmente. Os 10% restantes inseridos ao vivo no primário — o SELECT COUNT(*) na réplica refletiu as inserções instantaneamente, atestando a integridade.

Por que esquema estrela?

Otimizado para consultas OLAP. A tabela fato centraliza os eventos; as dimensões enriquecem a análise com contexto temporal, geográfico, demográfico e administrativo.

Surrogate Keys (SK)

Chaves substitutas evitam colisão de IDs ao integrar múltiplos datasets. Ex.: bairro com ID 100 em SP-cidade vira SK=1; mesmo ID de SP-estado vira SK=2.

14 / Esquema Dimensional
Como foi feito

Construção do Esquema Dimensional

Decisão de design

A Fato_Ocorrencias foi escolhida como centro por ser o objeto principal de estudo. Toda inteligência extraída pelos pesquisadores orbita em torno dos eventos criminais registrados.

Mapeamento OLTP → DW
Delegacia (OLTP) → Dim_Delegacia (DW)
Local_Ocorrencia → Dim_Local
Pessoa_Envolvida → Dim_Pessoa
data_ocorrencia → Dim_Tempo
Surrogate Key na prática
-- Dataset SP-Cidade
bairro_id = 100 → sk_local = 1

-- Novo dataset SP-Estado (integrado)
bairro_id = 100 → sk_local = 2

-- Sem colisão de chaves!
flag_vitima_fatal

Medida central da Fato: flag binária que permite calcular taxa de letalidade, comparar volumes e filtrar ocorrências críticas nas 9 consultas OLAP.

SUM(flag_vitima_fatal) COUNT(*) ROUND(..., 2)

16 / OLAP
Inteligência Analítica

Consultas OLAP

01
Regiões com maior criminalidade
Bairro + Cidade + Total de ocorrências. Direciona recursos de segurança pública.
02
Profissões mais vitimadas
Profissão + Total de vítimas. Subsidia policiamento estratégico por área de atuação.
03
Análise socioeconômica por cor/raça
Cor · Escolaridade · Profissão · Vítimas fatais. Identifica vulnerabilidade social.
04
Vítimas fatais por período
Série temporal por Ano/Mês. Detecta sazonalidade e valida políticas públicas.
05
Perfil de escolaridade
Grau de instrução × ocorrências. Correlaciona vulnerabilidade educacional e criminalidade.
06
Ranking de delegacias por volume
CTE Top 5 delegacias sobrecarregadas. Orienta alocação de policiais e recursos.
07
Letalidade por tipo de local
Taxa de letalidade % por ambiente. Prioriza prontidão de ambulâncias e viaturas.
08
Vulnerabilidade demográfica (Roll-up)
Faixa etária × gênero × mês. Slice & Dice para prevenção sazonal.
09
Criticidade por turno e localidade
Madrugada/Manhã/Tarde/Noite × bairro × letalidade. Dimensiona escalas de plantão.
